## 1. Автоматизація та інфраструктура розробки

### 🆕 [Makefile](file:///home/skmelyuk/Documents/projects/Dribbble/Dribble/Makefile)
* **Що зроблено**: Створено з нуля файл автоматизації завдань для проекту.
* **Чому це необхідно**: Дозволяє розробнику керувати всім життєвим циклом проекту (створення віртуального середовища, встановлення залежностей, запуск докера, застосування міграцій, запуск сервера та тестів) простими та уніфікованими командами, без необхідності пам'ятати складний синтаксис команд `docker compose exec...` та `python manage.py...`.

### 🆕 [pytest.ini](file:///home/skmelyuk/Documents/projects/Dribbble/Dribble/pytest.ini)
* **Що зроблено**: Створено конфігураційний файл тестування для pytest.
* **Чому це необхідно**: Дозволяє запускати тести через pytest без необхідності вручну прописувати змінну оточення `DJANGO_SETTINGS_MODULE` при кожному запуску. Він автоматично підключає налаштування проекту та оптимізує швидкість тестів за допомогою повторного використання бази даних.

### 🆕 [Dockerfile](file:///home/skmelyuk/Documents/projects/Dribbble/Dribble/Dockerfile)
* **Що зроблено**: Створено з нуля Docker-інструкцію для збірки контейнера веб-додатку.
* **Чому це необхідно**: Спочатку проект мав docker-compose лише для бази даних та Redis, а сам Django-сервер мав запускатися локально на хості. Dockerfile дозволяє запакувати бекенд у контейнер для зручного розгортання в єдиній локальній мережі з базою даних.
* **Важлива деталь**: Версію базового образу було змінено з `python:3.11-slim` на `python:3.12-slim`, оскільки проект використовує Django `6.0.6`, який вимагає Python не нижче версії `3.12`.

### 📝 [docker-compose.yml](file:///home/skmelyuk/Documents/projects/Dribbble/Dribble/docker-compose.yml)
* **Що зроблено**: Додано опис сервісу `web` (бекенд-контейнера) та пов'язано його з контейнерами `db` (PostgreSQL) та `redis` (Redis).
* **Чому це необхідно**: Об'єднує всі складові бекенду в єдину ізольовану мережу. Завдяки цьому контейнеру `web` більше не потрібен доступ до портів бази даних на локальному хості, а зв'язок відбувається через внутрішній DNS Docker по назвах сервісів (`db` та `redis`).

## 5. Усунення критичної помилки при копіюванні QueryDict з файлами

### 📝 [apps/shots/serializers.py](file:///home/skmelyuk/Documents/projects/Dribbble/Dribble/apps/shots/serializers.py)
* **Що зроблено**: Переписано метод `to_internal_value()`. Замість виклику глибокого копіювання `data.copy()` для об'єктів `QueryDict` реалізовано поверхневе копіювання (shallow copy) через ітерацію по списках `.lists()`.
* **Чому це необхідно**: При публікації роботи на сервер передається зображення (`image`). Об'єкт `QueryDict` у Django при виклику `.copy()` намагається зробити глибоку копію (`deepcopy`) всіх об'єктів, включно з відкритими файловими дескрипторами завантаженого зображення. Це викликало помилку `TypeError: cannot pickle 'BufferedRandom' instances` та повертало `500 Internal Server Error` при публікації нових робіт розробниками.
